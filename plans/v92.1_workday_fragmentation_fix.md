## Root cause

**`buildSessionsFromEvents()` treats a same-site, same-day duplicate ENTER (no accepted EXIT in between) as "prior enter never closed" and force-seals the open session as an incomplete no-finish fragment, then starts a new one.** v89's per-event replay loop ignored exactly this case (`Enter ignored — timer already running`); the v90 rewrite dropped that idempotency guard.

- **Code:** `www/index.html:2753-2765`, specifically line 2756:
  ```js
  if(ev.type==='enter'){
    if(open){ completed.push(open); open=null; } // prior enter never closed
  ```
  The behaviour is even documented as intended in the function's comment (`www/index.html:2690-2691`): *"Enter with a session already open → the open one is pushed as-is (incomplete, no finish) and a new one begins."* It was designed for the multi-**day** case (enter Monday, exit lost, enter Tuesday → must split) but is overbroad: it also fires for same-site same-day re-ENTERs, which on the rural Granite Belt are routine phantom events.
- **Introduced by:** commit `32f23a8` (v90 multi-session queue, 2026-07-01 21:10). The builder block is untouched since (verified via `git log -L`). The v89 guard it replaced is visible in `git show c7c3eef:www/index.html` (old `processGeoEvent()`): `if (activeDay()) { GeoLog.add('ignore', 'Enter ignored — timer already running…'); return; }`
- **What feeds it — the phantom ENTER stream (not new, present both days):** rural fused-location garbage fires spurious EXITs that are km off but get **rejected** by the v81 accuracy gate (acc 500–1300m > 150m). Each garbage EXIT still flips the GMS geofence's internal state to "outside", so when GPS reacquires a good fix inside the fence, a fresh DWELL ENTER fires — **with good accuracy, so it's ACCEPTED**. The accepted stream on a flutter day is therefore systematically `enter, enter, enter, …, exit` — the gate is structurally asymmetric (garbage exits rejected, clean re-enters accepted).
- **Why the 60s flutter-collapse didn't help:** `_processPendingGeoEventsInner()` (`www/index.html:8836-8854`) only collapses events within a 60-second burst. Today's accepted enters were ~90 min apart (10:15, 11:45, 13:15, 13:21) — hours-scale flutter sails straight through into the builder.

### Answers to the brief's five questions

1. **New-entry trigger:** an accepted native ENTER event during batch reconstruction (builder line 2755-2764), or live web/native start paths (those have guards and are not implicated).
2. **Finish trigger:** an accepted native EXIT paired to the open session (line 2766-2768); the trailing same-day exit additionally passes the v89 fresh-fix gate in `reconstructAndReconcile()` (line 3769-3774).
3. **Why no finish on the early entries:** their "finish" never existed — the builder *seals-incomplete on the next duplicate ENTER* instead of ignoring it. No exit was ever accepted mid-day (all garbage-rejected, correctly).
4. **Phantom starts:** yes — GMS re-arms the fence after each rejected garbage EXIT, then fires an accepted good-accuracy DWELL re-ENTER. Same pattern both days; only the replay code differed.
5. **Why the last entry has both times:** it's the only fragment whose open state survived until the one genuine accepted EXIT (13:28, when Steven actually left), so it got paired and passed the fresh-fix verify.

## Fix approach (minimal, surgical)

**Restore v89's duplicate-ENTER idempotency inside the pure builder, scoped to same-site + same-date.** In `buildSessionsFromEvents()` (`www/index.html:2755-2756`):

```js
if(ev.type==='enter'){
  if(open && open.site===ev.site && open.date===ev.date){
    // Same-site same-day re-ENTER with no accepted EXIT in between = still on
    // site (GMS re-armed the fence after a rejected garbage exit). v89
    // semantics: ignore. Surface it to the caller for GeoLog visibility.
    ignored.push(ev);
    continue;
  }
  if(open){ completed.push(open); open=null; } // prior enter never closed (different site or day)
  …
```

- Return shape becomes `{ completed, open, ignored }` — additive, existing callers unaffected.
- In `reconstructAndReconcile()` (`www/index.html:3736`), log each ignored event: `GeoLog.add('ignore','Enter ignored — session already open ('+ev.site+') at '+ev.time)` — parity with v89's field-proven log line. **Keep the builder pure** (no GeoLog/DB inside the `__V90_BUILDER_*__` markers — `test-tax.js`-style extraction in `test-sessions.js` depends on it); that's why ignored events are returned, not logged in place.
- Update the stale doc comment at `www/index.html:2690-2691`.

**What this preserves (the legitimate multi-entry capability):**
- Multi-**day**: enter Mon (exit lost) → enter Tue same site: dates differ → Mon still seals incomplete, Tue opens fresh. Unchanged.
- Multi-**site** same day: enter A → enter B: sites differ → A seals incomplete, B opens. Unchanged.
- Same-day re-entry after a **genuine accepted exit**: exit closed the session (open=null), so the next enter takes the existing merge-or-new-session path (≤90 min gap → merged with lunch banked; >90 min → second session). Unchanged.
- The only changed case is the pathological `enter → enter` same-site same-day with no accepted exit between — which is exactly the phantom pattern, and exactly what v89 ignored for a month of validated field use.

**Accepted trade-off (document, don't "fix"):** if Steven *genuinely* leaves and returns same-day but the departure EXIT gets accuracy-rejected, the day stays one continuous session (lunch gap not auto-banked). That is v89's exact behaviour, it's what the review/Adjust step exists for, and it's the right asymmetry — a missed split is one Adjust; a phantom split corrupts the day.

**Do NOT fix it elsewhere:** don't widen `BURST_MS` (flutter spans hours; a wide window would cancel real enter/exit pairs), don't tighten/loosen the 150m accuracy gate (the re-enters are triggered by genuinely good fixes — rejecting them breaks real arrival detection), don't touch the Java capture layer (unchanged since v89 and field-validated).

**Size: small** — ~6 lines in the builder + ~3 lines of caller logging + comment update + tests. Single file (`www/index.html`) plus `test-sessions.js` cases. Bump `APP_VERSION` (point release, e.g. `v101.2`).

### Secondary findings (optional, fixer's discretion — separate concerns, tiny)

- **S1 — dead post-stop flag check:** `_confirmDepartureThenStop()._commit()` (`www/index.html:3584-3589`) checks `activeDay().finish` *after* `autoStopTimer()`, but v90's seal-and-clear (line 3695-3696) makes `activeDay()` null there — so `geoAutoStopTriggered` is never set and a misleading `…flag NOT set` ignore-line is logged on **every successful reconstructed stop** (visible in today's log at 13:31:30). Harmless in practice (activeDay is null, stop paths no-op on it) but noisy and stale. Suggested cleanup: have `autoStopTimer()` return a success boolean and log from that; do not resurrect the stop-flag semantics (v90's `_resetGeoTriggerFlags()` deliberately re-arms for multi-job days).
- **S2 — GeoLog UTC date:** `GeoLog.add` sets `date` from `toISOString()` (UTC) while `time` is local (`www/index.html:3232-3234`) — Brisbane entries before 10:00 land on the *previous* day's mirror doc (today's 07:06–08:52 arrival block sits in `geolog/2026-07-01`, which cost diagnosis time). One-line fix: `date: todayStr()`. Session records are unaffected (`todayStr()` at line 3059 is already local).

## Risk to existing records

**No migration needed.** The broken rows live only in `mcn_unconfirmed` (review backlog) — they are structurally non-billable (invoice/stats read `mcn_days` only) and will never be counted unless explicitly Confirmed. The bug produces no further fragments for 2 Jul (the native queue was drained). Cleanup = **manual, via the existing review UI**: Reject the no-finish fragments, Adjust one entry to the real 08:45–13:30 and Confirm (or Reject all + Manual Entry). Rejected rows are retained as audit records by design.

Do **not** ship an auto-repair migration: it would need heuristics to distinguish phantom fragments from legitimate incomplete sessions (e.g. a real multi-day open Monday session), for one day's worth of one user's reviewable data. Note for the fixer: until the fix ships, **every future flutter day will fragment the same way** — ship the fix before the next field day, or warn Steven to expect fragments in the backlog and Reject them.
